# DuckPools Bankroll Database Schema

## Overview

This document describes the PostgreSQL database schema for the DuckPools bankroll management system. The database stores bankroll state, transactions, alerts, auto-reload events, and risk projections.

## Requirements

- PostgreSQL 15+
- Python 3.12+
- Dependencies listed in `requirements.txt` (updated with database packages)

## Database Setup

### 1. Create Database

```bash
# Connect to PostgreSQL as superuser
psql -U postgres

# Create database and user
CREATE DATABASE duckpools;
CREATE USER duckpools_user WITH PASSWORD 'your_secure_password';
GRANT ALL PRIVILEGES ON DATABASE duckpools TO duckpools_user;
\q
```

### 2. Configure Environment Variables

Create a `.env` file in the backend directory:

```env
# Database
DATABASE_URL=postgresql+asyncpg://duckpools_user:your_secure_password@localhost:5432/duckpools
```

### 3. Run Migrations

```bash
cd backend

# Install dependencies
pip install -r requirements.txt

# Run all migrations
alembic upgrade head

# Verify tables created
psql -U duckpools_user -d duckpools -c "\dt"
```

## Schema Tables

### bankroll_state

Current state of the house bankroll. This table stores a single row representing the real-time bankroll status.

**Key Columns:**
- `current_balance`: Current bankroll balance in ERG
- `total_tv`: Total value (liquidity + bankroll)
- `house_liquidity`: House-owned liquidity
- `lp_pool_balance`: Liquidity provider pool balance
- `max_payout_capacity`: Maximum payout the bankroll can handle
- `auto_reload_enabled`: Whether auto-reload is active
- `min_balance_threshold`: Trigger balance for auto-reload
- `reload_amount`: Amount to reload when threshold triggered
- `risk_of_ruin`: Calculated risk of ruin probability
- `kelly_fraction`: Kelly criterion optimal bet fraction

### bankroll_transactions

Immutable audit trail of all bankroll movements.

**Key Columns:**
- `tx_type`: Transaction type (deposit, withdrawal, bet_placed, bet_won, bet_lost, etc.)
- `amount`: Transaction amount (positive for incoming, negative for outgoing)
- `balance_before`: Balance before transaction
- `balance_after`: Balance after transaction
- `tx_hash`: On-chain transaction hash (if applicable)
- `bet_id`: Game bet identifier (for bet-related transactions)
- `game_type`: Game type (coinflip, dice, plinko)

### bankroll_alerts

Historical record of bankroll alerts generated by the monitoring system.

**Key Columns:**
- `alert_type`: Type of alert (low_balance, unusual_loss, auto_reload_triggered, etc.)
- `severity`: Severity level (info, warning, error, critical)
- `balance_at_alert`: Bankroll balance when alert was triggered
- `threshold_value`: Threshold value that was crossed
- `acknowledged`: Whether alert has been acknowledged
- `resolved`: Whether alert has been resolved

### auto_reload_events

Historical record of automatic bankroll reload events.

**Key Columns:**
- `success`: Whether the reload succeeded
- `reload_amount`: Amount reloaded
- `trigger_reason`: Reason for reload (low_balance, etc.)
- `balance_at_trigger`: Balance when reload was triggered
- `tx_hash`: On-chain transaction hash
- `error_message`: Error message if reload failed

### risk_projections

Variance and risk projections based on historical data.

**Key Columns:**
- `current_balance`: Balance at time of calculation
- `variance`: Variance of bankroll changes
- `standard_deviation`: Standard deviation
- `p1_balance, p5_balance, ... p99_balance`: Percentile projections
- `kelly_fraction`: Kelly criterion optimal fraction
- `risk_of_ruin`: Risk of ruin probability
- `bankroll_multiple`: Bankroll / standard deviation ratio

## Common Queries

### Get Current Bankroll State

```sql
SELECT * FROM bankroll_state;
```

### Get Recent Transactions

```sql
SELECT * FROM bankroll_transactions
ORDER BY created_at DESC
LIMIT 50;
```

### Get Unresolved Alerts

```sql
SELECT * FROM bankroll_alerts
WHERE resolved = false
ORDER BY created_at DESC;
```

### Get Recent Auto-Reload Events

```sql
SELECT * FROM auto_reload_events
ORDER BY created_at DESC
LIMIT 20;
```

### Calculate Total House Edge Revenue

```sql
SELECT SUM(amount) as total_revenue
FROM bankroll_transactions
WHERE tx_type = 'house_edge';
```

## Maintenance

### Backup

```bash
# Full backup
pg_dump -U duckpools_user -d duckpools > duckpools_backup.sql

# Restore
psql -U duckpools_user -d duckpools < duckpools_backup.sql
```

### Monitoring Queries

```sql
-- Check database size
SELECT pg_size_pretty(pg_database_size('duckpools'));

-- Check table sizes
SELECT 
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS size
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC;
```

## Security Notes

1. **Never commit credentials**: Database passwords should be in environment variables, not in code
2. **Use prepared statements**: The ORM handles this automatically
3. **Regular backups**: Schedule daily backups of the database
4. **Access control**: Only application servers should have write access to the database

## Performance Considerations

1. **Indexes**: All foreign key columns and frequently queried columns are indexed
2. **Connection pooling**: SQLAlchemy async engine with pool_size=10, max_overflow=20
3. **Time zone aware**: All timestamps are stored with timezone information
4. **Decimal precision**: Monetary values use Numeric(36, 18) for high precision

## Troubleshooting

### Migration Fails

```bash
# Check current revision
alembic current

# Check if migration was partially applied
SELECT * FROM alembic_version;
```

### Lock Issues

```bash
# Check for blocking locks
SELECT 
    pid,
    usename,
    pg_blocking_pids(pid) as blocked_by,
    query as blocked_query
FROM pg_stat_activity
WHERE cardinality(pg_blocking_pids(pid)) > 0;
```

### Connection Pool Exhaustion

Increase `pool_size` and `max_overflow` in `app/db/__init__.py` if you see connection errors.
